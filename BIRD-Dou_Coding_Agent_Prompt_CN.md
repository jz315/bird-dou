# BIRD-Dou 编码 AI 总提示词

你正在实现一个研究级斗地主 AI 项目 **BIRD-Dou**。

开始工作前，必须完整阅读用户提供的《BIRD-Dou 斗地主 AI 研发实施规格书》，并把它视为项目的最高规格来源。不要一次性实现全部系统，严格按照里程碑和 Ticket 顺序推进。

## 总目标

构建一个：

- 规则完全正确；
- 能复现 DouZero；
- 使用结构化点数、历史和候选动作编码；
- 使用严格牌数守恒的隐藏牌 Belief CRF；
- 使用训练期完整信息 Teacher，但执行期不读取隐藏牌；
- 使用信息集一致蒸馏；
- 对地主、上家农民、下家农民进行角色适配；
- 支持 DMC、V-trace 和 Hybrid 对照；
- 最终支持叫牌、出牌、完整计分、统一评测和残局搜索；

的可复现斗地主 AI。

## 不可违反的约束

1. 规则正确性优先于模型开发。
2. 未完成 DouZero differential test 前，不得开始高级模型。
3. Student Observation 绝不能包含真实对手或队友手牌。
4. 完整手牌只允许用于环境内部、监督标签、Teacher 和 Privileged Critic。
5. 所有规则差异由 `RuleConfig` 控制，禁止静默写死某个平台规则。
6. 每次只实现一个 Ticket，不得顺手大规模重构。
7. 所有公共接口必须有类型、文档、错误处理和测试。
8. 不得提交 TODO 式空实现、固定返回值、伪代码冒充实现或只通过单例测试的代码。
9. 每次修改后运行格式化、静态检查、单元测试和相关 benchmark。
10. 若规格、现有代码和测试存在冲突，先报告冲突，不得自行猜测并改变规则。
11. 新模型、新损失、新特征必须带配置开关，以便消融和回退。
12. 不得声称“更强”或“更快”，除非提供同条件评测或 benchmark。

## 每个 Ticket 的固定工作流程

### 开工前

先输出：

1. 本 Ticket 的目标；
2. 依赖是否已经满足；
3. 将阅读的文件；
4. 将修改/新增的文件；
5. 公共 API 和数据结构变化；
6. 测试计划；
7. 可能风险。

### 实现中

- 保持改动最小；
- 不改变无关模块；
- 为边界情况写测试；
- 规则模块优先使用 property-based test；
- 性能模块保留可比较的 benchmark；
- 任何随机行为必须可以由 seed 复现。

### 完成后

输出：

1. 实现摘要；
2. 修改文件清单；
3. 关键设计选择；
4. 测试命令及结果；
5. benchmark 结果；
6. 尚未解决的问题；
7. 当前 Ticket 是否达到规格书验收条件；
8. 下一 Ticket 是否可以开始。
